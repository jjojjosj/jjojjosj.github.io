LLM(대규모 언어 모델)을 활용한 개발이 보편화되면서 이를 둘러싼 여러 엔지니어링 개념들이 등장했습니다. 그 중 **프롬프트 엔지니어링(Prompt Engineering)**, **컨텍스트 엔지니어링(Context Engineering)**, **하네스 엔지니어링(Harness Engineering)** 은 혼용되는 경우가 많지만, 실제로는 각기 다른 계층과 범위를 다루는 개념입니다.

이 세 가지 개념의 차이를 명확히 이해하면, LLM 기반 시스템을 설계할 때 어느 층위에서 문제를 해결해야 하는지 훨씬 빠르게 판단할 수 있습니다.

---

## 프롬프트 엔지니어링 (Prompt Engineering)

### 정의

프롬프트 엔지니어링은 **LLM에 전달하는 텍스트 입력(프롬프트) 자체를 최적화하는 기술**입니다. 모델이 원하는 출력을 생성하도록 지시문의 표현, 구조, 순서를 조정하는 데 집중합니다.

### 핵심 기법

- **Zero-shot / Few-shot Prompting**: 예시 없이 혹은 몇 가지 예시를 제공하여 태스크를 지시
- **Chain-of-Thought (CoT)**: "단계적으로 생각해봐"처럼 추론 과정을 명시적으로 유도
- **Role Prompting**: "당신은 시니어 DevOps 엔지니어입니다"와 같이 역할을 부여
- **Instruction Tuning 스타일**: 명령형으로 구체적이고 명확하게 작성

### 적용 범위

프롬프트 엔지니어링의 범위는 기본적으로 **단일 LLM 호출 내의 텍스트**에 국한됩니다. 어떤 단어를 쓰느냐, 어떤 순서로 지시하느냐, 어떤 형식을 요구하느냐의 문제입니다.

### 예시

```
# Bad
"코드 설명해줘"

# Good (프롬프트 엔지니어링 적용)
"아래 Python 코드를 분석하고, 주니어 개발자가 이해할 수 있도록
각 함수의 역할과 잠재적 버그를 한국어로 설명해줘.
출력 형식은 함수명 → 설명 → 주의사항 순으로 작성해줘."
```

> 프롬프트 엔지니어링은 LLM을 다루는 가장 기초적인 기술이지만, **단일 호출 최적화**라는 한계가 명확합니다.

## 컨텍스트 엔지니어링 (Context Engineering)

### 정의

컨텍스트 엔지니어링은 **LLM의 컨텍스트 윈도우(Context Window)에 무엇을 넣을지를 설계하는 기술**입니다. 단순히 프롬프트 문장을 다듬는 것을 넘어, 모델이 정확한 답을 내리기 위해 필요한 **모든 정보를 선별·구성·압축**하는 작업입니다.

### 프롬프트 엔지니어링과의 차이

| 구분      | 프롬프트 엔지니어링  | 컨텍스트 엔지니어링                         |
| --------- | -------------------- | ------------------------------------------- |
| 관심사    | 지시문의 표현과 구조 | 컨텍스트 윈도우 전체의 정보 구성            |
| 범위      | 단일 프롬프트 텍스트 | 시스템 프롬프트 + 히스토리 + 문서 + 툴 결과 |
| 핵심 질문 | "어떻게 말할까?"     | "무엇을 보여줄까?"                          |

### 컨텍스트 윈도우를 구성하는 요소들

- **시스템 프롬프트**: 역할, 행동 지침, 제약 조건
- **대화 히스토리**: 이전 턴의 메시지 (얼마나 유지할지 관리 필요)
- **검색된 문서 (RAG)**: 외부 지식베이스에서 가져온 관련 청크
- **툴 실행 결과**: 함수 호출, API 응답, 코드 실행 출력
- **메모리**: 장기 기억 시스템에서 불러온 요약 정보

### 실제 적용 사례

RAG(Retrieval-Augmented Generation) 시스템을 구축할 때, "어떤 문서를 검색해서 컨텍스트에 포함시킬 것인가"를 결정하는 작업이 대표적인 컨텍스트 엔지니어링입니다. 단순히 가장 유사한 문서 TOP-5를 넣는 게 아니라:

- 중복 정보 제거
- 최신성 가중치 적용
- 토큰 예산 내에서 가장 유의미한 청크 선택
- 컨텍스트 내 배치 순서 최적화 (Positional Bias 고려)

이러한 판단들이 모두 컨텍스트 엔지니어링의 영역입니다.

> **컨텍스트 윈도우는 LLM의 단기 기억**입니다. 여기에 무엇을, 얼마나, 어떤 순서로 넣느냐가 모델 성능에 직접적인 영향을 미칩니다.

## 하네스 엔지니어링 (Harness Engineering)

### 정의

하네스 엔지니어링은 **LLM 호출을 감싸는 인프라, 오케스트레이션, 실행 프레임워크를 설계하는 기술**입니다. "하네스(harness)"는 말 그대로 LLM을 제어·구동하기 위한 **안장 장치(scaffolding)** 를 의미합니다.

프롬프트나 컨텍스트가 "모델에게 무엇을 전달하느냐"의 문제라면, 하네스는 **"모델 호출 전후로 어떤 일이 일어나느냐"** 를 다룹니다.

### 하네스가 담당하는 것들

- **에이전트 루프 (Agent Loop)**: 모델이 툴을 호출하고 결과를 받아 다시 추론하는 반복 사이클 관리
- **툴/함수 실행 레이어**: 모델이 요청한 도구(웹 검색, 코드 실행, 파일 읽기 등)를 실제로 실행
- **상태 관리 (State Machine)**: 멀티스텝 태스크에서 현재 진행 상태 추적
- **오류 처리 및 재시도**: API 실패, 할루시네이션 감지, 타임아웃 처리
- **훅 시스템 (Hooks)**: 특정 이벤트(툴 호출 전/후, 메시지 전송 전 등)에 자동으로 실행되는 사이드이펙트
- **멀티 에이전트 조율**: 여러 LLM 에이전트가 협력하는 파이프라인 구성
- **권한 및 샌드박싱**: 에이전트가 실행할 수 있는 작업의 범위를 제한

### 실제 예시: Claude Code의 하네스

Claude Code를 예로 들면, 사용자가 메시지를 입력했을 때:

1. **Pre-submit hook**: 메시지 전송 전 특정 검사 수행
2. **LLM 호출**: 프롬프트 + 컨텍스트를 모델에 전달
3. **툴 승인 레이어**: 파일 수정, 명령 실행 등에 대한 권한 체크
4. **툴 실행**: Bash, Edit, Read 등의 실제 실행
5. **Post-tool hook**: 툴 실행 후 자동화된 검사나 알림
6. **결과 피드백**: 툴 결과를 다시 컨텍스트에 포함하여 다음 추론 진행

이 전체 파이프라인을 설계하고 관리하는 것이 하네스 엔지니어링입니다.

```
# 하네스 엔지니어링 관점의 에이전트 루프 (의사코드)
while not task_complete:
    run_pre_hooks(event="before_llm_call")
    response = llm.call(context_window)

    if response.has_tool_call:
        permission = check_permission(response.tool)
        if permission.granted:
            result = execute_tool(response.tool)
            run_post_hooks(event="after_tool_call", result=result)
            context_window.add(result)
        else:
            context_window.add("Permission denied")
    else:
        task_complete = True
```

## 세 가지 엔지니어링의 계층 구조

```
┌─────────────────────────────────────────┐
│          하네스 엔지니어링               │
│  (에이전트 루프, 툴 실행, 오케스트레이션) │
│  ┌───────────────────────────────────┐  │
│  │       컨텍스트 엔지니어링          │  │
│  │  (무엇을 컨텍스트 윈도우에 넣을지)  │  │
│  │  ┌─────────────────────────────┐  │  │
│  │  │     프롬프트 엔지니어링       │  │  │
│  │  │  (어떻게 지시문을 표현할지)   │  │  │
│  │  └─────────────────────────────┘  │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

세 개념은 **중첩된 계층 구조**를 이룹니다. 프롬프트 엔지니어링이 가장 안쪽의 미시적 층위, 컨텍스트 엔지니어링이 중간 층위, 하네스 엔지니어링이 가장 바깥쪽의 거시적 층위입니다.

## AI 개발 워크플로우에서의 역할 분담

실제 LLM 기반 시스템을 구축할 때 각 개념이 어떤 문제를 해결하는지 정리하면 다음과 같습니다.

| 문제 상황                           | 적용 영역                           |
| ----------------------------------- | ----------------------------------- |
| 모델이 원하는 포맷으로 답변을 안 함 | 프롬프트 엔지니어링                 |
| 모델이 최신 정보를 모름             | 컨텍스트 엔지니어링 (RAG)           |
| 긴 대화에서 앞 내용을 잊어버림      | 컨텍스트 엔지니어링 (히스토리 관리) |
| 에이전트가 무한 루프에 빠짐         | 하네스 엔지니어링                   |
| 툴 실행 결과를 다음 추론에 반영     | 하네스 엔지니어링                   |
| 민감한 파일 접근을 막아야 함        | 하네스 엔지니어링 (권한 레이어)     |
| 여러 에이전트가 협업하게 하려면     | 하네스 엔지니어링 (멀티에이전트)    |

## 마치며

LLM을 단순히 API로 호출하는 수준을 넘어 **신뢰할 수 있는 프로덕션 시스템**으로 만들기 위해서는 세 가지 엔지니어링을 모두 다룰 줄 알아야 합니다.

- 좋은 **프롬프트 엔지니어링**은 모델의 잠재력을 끌어냅니다.
- 잘 설계된 **컨텍스트 엔지니어링**은 모델이 올바른 정보를 바탕으로 판단하게 합니다.
- 견고한 **하네스 엔지니어링**은 시스템이 예측 가능하게 동작하고 실패해도 안전하게 복구되도록 만듭니다.

특히 인프라 엔지니어 관점에서는 하네스 엔지니어링이 기존의 서비스 메시, 오케스트레이션, 옵저버빌리티 개념과 맞닿아 있어 매우 익숙한 영역일 것입니다. LLM을 하나의 외부 서비스처럼 바라보고, 그 앞뒤로 신뢰할 수 있는 인프라를 설계하는 시각이 앞으로 더욱 중요해질 것입니다.
